# Design: add-codex-support

## Design Items

### DS-1: Codex Skills and Provider Registration
- **Refs**: PR-1
- **Source**: PR-1 (proposal.md)
- **Responsibility**: Register `codex` and generate the ten project-scoped Codex Skills from the shared workflow registry.
- **Key Interfaces**: `registerCodexProvider()`, `generateCodexSkills(config)`, `generateCodexSkill(step)`.

#### Detailed Design
The provider id is `codex`, display name `OpenAI Codex CLI`, and `supportsCommands: false`. Generation returns ten deterministic descriptors at `.agents/skills/bp-<step>/SKILL.md` for `init`, `roadmap`, `propose`, `plan`, `apply`, `review`, `archive`, `continue`, `ff`, and `loop`. Each file uses Codex frontmatter with `name: bp:<step>` and a one-line description; it MUST omit OMP/Agent-only `hide` and `argument-hint` fields. The body is sourced from the corresponding `WORKFLOW_REGISTRY` entry, preserving existing workflow semantics. Missing registry entries use a deterministic fallback body. Provider registration is idempotent and unknown configured platform ids continue to fail through the existing registry error path.

### DS-2: Codex Hook Surface and Deterministic Runtime
- **Refs**: PR-1, PR-3
- **Source**: PR-1 (proposal.md), PR-3 (proposal.md)
- **Responsibility**: Generate Codex hook configuration and the compiled handler template implementing bp runtime parity.
- **Key Interfaces**: `generateCodexHooks(config)`, `generateCodexHandler(config)`, `CODEX_HOOK_EVENTS`.

#### Detailed Design
Generate `.codex/hooks.json` with the Codex schema and exactly five events: `SessionStart`, `SessionStop`, `UserPromptSubmit`, `PreToolUse` (matcher `Bash`), and `PostToolUse` (matcher `Bash`). Every command invokes `node .codex/hooks/bp-handler.mjs <event-argument>`. Generate the handler from `src/templates/codex/handler.tmpl.ts`; compilation is performed by the update/generation path and output is byte-deterministic for the same config. The handler maps session start to compact bp context, prompt/tool lifecycle events to workflow-state reinjection, and stop to a no-op/terminal notification while preserving the OMP contract. It exits successfully when `bp/config.yaml` is absent or hooks are disabled, and returns a non-zero/error payload only for malformed hook input or failed bp runtime invocation; it MUST never overwrite unrelated `.codex` files.

### DS-3: Lifecycle Touchpoints and Dispatch
- **Refs**: PR-2
- **Source**: PR-2 (proposal.md)
- **Responsibility**: Make Codex selectable, generated, cleaned, and represented in dispatch instructions.
- **Key Interfaces**: init wizard platform option; update stale-file cleanup; `bp dispatch` Codex format.

#### Detailed Design
Add Codex to the interactive multi-select with a description naming `.agents/skills/` and `.codex/hooks.json`; retain existing defaults for `--yes` and fallback mode. Add `.codex/` and `.agents/` to generated gitignore entries. Cleanup removes only generated `.codex/hooks.json` and directories matching `.agents/skills/bp-*` when absent from the current generation set; arbitrary files under `.codex` and non-bp skills remain untouched. Dispatch uses isolation type `none`, explicitly instructing the orchestrator to create a temporary worktree with `git worktree add`; format uses `tool: task` and documents that the Codex MCP adapter may override the tool name. Config/template platform validation accepts `codex`.

### DS-4: Documentation and Contract Delta
- **Refs**: PR-3
- **Source**: PR-3 (proposal.md)
- **Responsibility**: Document Codex parity and publish the behavioral contract for archive merge.
- **Key Interfaces**: Codex rows/section in `README.md`, `AGENTS.md`, and `docs/platform-integration.md`; delta spec.

#### Detailed Design
Documentation describes supported Codex v0.140+ Skills layout, hook event mapping, generated files, update cleanup boundaries, and worktree dispatch behavior. The delta spec adds one complete `codex-platform-support` requirement with happy, edge, and error scenarios and modifies the existing three-platform generation requirement to four platforms without changing existing outputs.

## Architecture Decisions

### D-1: Use `.agents/skills` as the canonical Skills location
- **Status**: ACCEPTED
- **Decision**: Generate one directory per workflow skill under `.agents/skills/bp-<step>/SKILL.md` with colon slash-command names (`bp:<step>`).
- **Reason**: Codex v0.140+ project-scoped Skills use this path and the requested namespace matches OMP invocation semantics.
- **Alternatives**: `.codex/skills` is non-canonical; flat `.md` files cannot carry Codex Skill metadata or directory contract.

### D-2: Single generated handler template
- **Status**: ACCEPTED
- **Decision**: Compile one TypeScript handler template and dispatch event arguments from `hooks.json`.
- **Reason**: Centralizes parity logic, keeps generated bytes deterministic, and avoids five divergent scripts.
- **Alternatives**: Five scripts duplicate runtime behavior; direct shell commands cannot implement context injection and error handling.

### D-3: Worktree orchestration for Codex isolation
- **Status**: ACCEPTED
- **Decision**: Set dispatch isolation type to `none` and instruct the orchestrator to run `git worktree add`.
- **Reason**: Codex has `--cd` but no native isolated-worktree primitive in the supported contract.
- **Alternatives**: Claiming native isolation would be incorrect; sharing the current directory risks concurrent executor interference.

### D-4: Conservative update cleanup
- **Status**: ACCEPTED
- **Decision**: Remove only generated hooks and `bp-*` Skill directories.
- **Reason**: Users may keep unrelated Codex configuration and Skills in the same directories.
- **Alternatives**: Deleting `.codex` or `.agents` wholesale is destructive and violates update safety.

## Technical Approach

### Architecture Diagram
```text
[EXISTING] bp init/update ──> [MODIFIED] generators/index.ts ──> [NEW] CodexProvider
                                  │                                      ├──> [NEW] Skills (10)
                                  │                                      └──> [NEW] hooks.json + handler.mjs
[EXISTING] WORKFLOW_REGISTRY ────────────────────────────────────────────┘
[MODIFIED] init wizard / gitignore / update cleanup / dispatch ──> generated project files
[MODIFIED] docs + specs ──> user contract and archive delta merge
```

### Core Data Structures
```typescript
interface GeneratedFile { path: string; content: string }
interface CodexSkillDefinition { step: string; name: `bp:${string}`; description: string }
interface CodexHookConfig { $schema: string; hooks: Record<string, Array<{ matcher?: string; hooks: Array<{ type: 'command'; command: string }> }>> }
```

### Data Flow
1. `bp init` or `bp update` loads `ProjectConfig.platform` and `generateAll` resolves `codex`.
2. The provider renders ten workflow bodies and hook configuration plus compiled handler descriptors.
3. `writeGeneratedFiles` writes files; update cleanup compares generated paths and removes only stale Codex-owned entries.
4. Codex invokes handler commands; the handler reads bp config/state and emits compact context/workflow-state payloads.
5. `bp dispatch` selects Codex isolation/format records and prints worktree-based executor instructions.

### Interface Design
#### `generateCodexSkills(config)`
- **Request**: `ProjectConfig`.
- **Response 200**: deterministic `GeneratedFile[]` containing ten `.agents/skills/bp-*/SKILL.md` files.
- **Error**: malformed config or missing workflow source throws a validation/config error; no partial file write occurs.
- **Source**: `specs/platform-gen/spec.md#codex-platform-support`

#### `generateCodexHooks(config)`
- **Request**: `ProjectConfig`.
- **Response 200**: descriptors for `.codex/hooks.json` and `.codex/hooks/bp-handler.mjs`.
- **Error**: invalid configuration fails generation with exit code 1; existing unrelated `.codex` files remain unchanged.
- **Source**: `specs/platform-gen/spec.md#codex-platform-support`

#### `bp dispatch executor --change <name>` for Codex
- **Request**: configured Codex platform and optional change name.
- **Response 200**: instructions naming `git worktree add`, `tool: task`, and executor artifact contract.
- **Error**: unknown role/platform/change prints clean error and exits 1.
- **Source**: `specs/platform-gen/spec.md#codex-dispatch-isolation`

- `PlatformRegistry.generateAll` and platform config validation now accept one additional id.
- Dispatch templates consume the new isolation/format records.

### Test Impacts
- Existing generator, init, update, dispatch, and lifecycle tests require Codex cases; existing platform snapshots MUST remain unchanged.

## TDD Strategy
- **behavior tasks**: RED -> GREEN -> REFACTOR (3 commits per task)
- **config/scaffolding/docs**: direct implementation (1 commit per task)
- **refactor**: verify tests pass -> refactor -> verify again

Use deterministic temp directories for lifecycle tests. Snapshot generated descriptors, assert exact event keys/matchers, and test cleanup preservation of unrelated files. Handler tests invoke generated source with stubbed bp commands and verify disabled/missing-config behavior.

## Risks
| Risk | Impact | Likelihood | Mitigation |
|---|---|---|---|
| Codex hook protocol changes | Hooks fail at runtime | Medium | Pin documented v0.140+ schema, snapshot exact JSON, isolate handler arguments. |
| Handler compilation differs by environment | Non-deterministic generated bytes | Medium | Compile one checked-in template with fixed options and test two invocations for byte equality. |
| Cleanup removes user files | Data loss | Low | Match only exact generated hooks path and `bp-*` skill directories; add preservation tests. |
| `task` MCP name differs in Codex deployment | Dispatch cannot launch sub-agent | Medium | Print explicit override guidance and keep dispatch table isolated for future adapter update. |
