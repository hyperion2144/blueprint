# Proposal: refactor-templates-i18n

## Intent

The current specwf command and skill templates have five structural problems that hurt agent comprehension and maintainability:

1. **Non-standard template format**: Commands and skills use ad-hoc sections (概述, 前置条件, 执行步骤) instead of the industry-standard OpenSpec format (Input → Steps → Output pattern). This makes templates harder for agents to parse and execute.

2. **Duplicated content across command/skill pairs**: Each workflow step has a `.omp/commands/specwf-<step>.md` and `.omp/skills/specwf-<step>/SKILL.md` with manually duplicated content. These should be the same content with different formatting wrappers — the command variant for command-supporting agents, the skill variant for skill-only agents.

3. **Chinese-language content**: All template files, skill files, command files, and generated documentation are written in Chinese. AI coding agents (the primary consumers of these files) understand English more precisely, leading to misinterpretation and lower-quality execution.

4. **Continue CLI returns file paths, not content**: `specwf continue` outputs "下一步: plan" plus a file path reference (`.omp/commands/specwf-plan.md`), requiring the agent to do an extra read before it can execute. The continue command should output the full next-step instructions inline.

5. **Templates in markdown files instead of TypeScript**: Templates live as `.md` files under `src/public/templates/` loaded at runtime via `readFileSync`. Following OpenSpec's pattern (see `onboard.ts`), templates should be TypeScript constants/functions within the source code, enabling type-safety, IDE autocompletion, and single-source-of-truth for both command and skill generation.

## Scope

### In scope

- Rewrite all 16 command templates from Chinese to English, following OpenSpec's Input/Steps/Output structure
- Unify command and skill generation from a single TypeScript source-of-truth (one definition → two output formats)
- Refactor `specwf continue` CLI to inline the next step's full command content instead of a file path reference
- Move all templates from markdown files under `src/public/templates/` into TypeScript modules under `src/templates/`
- Update all agent definitions (`.omp/agents/`) to English
- Update all output artifact templates (proposal, design, tasks, etc.) to English
- Ensure CLI `template` command reads from new TypeScript template sources
- Regenerate all `.omp/commands/` and `.omp/skills/` output files

### Out of scope

- Changing the agent orchestration logic (how agents are spawned and managed)
- Adding new workflow steps or removing existing ones
- Changing the state machine transitions
- Modifying the project.yml configuration format
- Adding support for new platforms (claude-code, etc.)

## Approach

### Architecture direction

Follow OpenSpec's `SkillTemplate` / `CommandTemplate` pattern from `src/core/templates/types.ts`:

```typescript
interface SkillTemplate {
  name: string;
  description: string;
  instructions: string;
}

interface CommandTemplate {
  name: string;
  description: string;
  category: string;
  tags: string[];
  content: string;  // same as SkillTemplate.instructions
}
```

One TypeScript module per workflow step exports both templates from the same `instructions` source string:
- `src/templates/workflows/apply.ts` → `getApplySkillTemplate()` + `getApplyCommandTemplate()`
- `src/templates/workflows/plan.ts` → `getPlanSkillTemplate()` + `getPlanCommandTemplate()`
- etc.

### Template format

Each instruction follows the OpenSpec pattern:

```
## Input

- Required files and their expected content
- State/context that must be available before starting

## Steps

### Step 1: ...
### Step 2: ...

## Output

- Files produced
- State changes made

## Guardrails

- Common pitfalls
- Hard constraints
```

### Continue CLI change

Instead of:
```
→ 下一步: plan
   Slash 命令: /specwf:plan
   参考: .omp/commands/specwf-plan.md
```

Output the full command body inline so the agent can execute without another file read.

### Compatibility

- Existing `.omp/commands/` and `.omp/skills/` files are regenerated by `specwf update`
- Output artifact templates remain in the same filesystem locations; only the source changes
- No API changes to the CLI commands; `specwf template <type>` continues to work

## Must-haves

- All 16 command/skill pairs generated from a single TypeScript source per step
- Every template and generated file is in English
- `specwf continue` outputs the next step's full instructions inline
- Zero markdown template files remain under `src/public/templates/`
- `specwf update` regenerates all output files correctly
- Existing integration tests pass after regeneration

## Non-goals

- Not changing the number or identity of workflow steps
- Not adding i18n/l10n framework for future multi-language support
- Not changing agent system prompts beyond language translation
- Not modifying the `project.yml` schema
