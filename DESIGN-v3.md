# Blueprint v2.1 深化优化方案：从静态流程到自适应系统

**版本**：v1.0  
**日期**：2026-07-22  
**基于版本**：Blueprint v2.1 (model-enhanced workflow redesign)  

---

## 核心判断

v2.1 已经完成了「**按风险定价流程强度**」的第一层进化——四级分级、预算熔断、模型分档、prompt 精简，全部到位。

下一层进化方向是：**从人设计的静态流程，进化为数据驱动的动态系统**。

- 工程约束型能力继续加厚（spec 治理、依赖管理、质量门控）
- 能力补偿型能力继续变薄（自动精简、自动降级、自动校准）
- 新增：系统自己观察、自己学习、自己调整参数的闭环能力

---

## 一、自适应闭环：从人工调参到系统自优化

v2.1 的分级阈值、预算参数、降级策略都是静态配置，靠人拍脑袋调。积累足够多的 `.meta/` 数据后，框架可以自己学习优化。

### 1.1 分级阈值自动校准

**现状问题**：默认阈值（20 个 task 算大、3 轮 review 算多）是通用值，不同团队/项目差异很大。

**优化方案**：
- 初始阶段使用默认阈值
- 跑过 20 个 change 后，自动用项目中位数重新校准告警线
- 某团队 change 普遍 30+ task → 系统自动放宽，不瞎喊狼来了
- 校准结果写入 `config.yaml`，可人工覆盖

### 1.2 降级成功率反馈环

**现状问题**：哪些角色能降级、哪些不能，靠经验判断。

**优化方案**：
- 统计「自动降级后出现 BLOCKER 的概率」
- 某角色降级成功率 > 90% → 扩大降级范围（更多场景自动降）
- 某角色降级成功率 < 60% → 收紧甚至关闭自动降级
- 降级失败记录写入 `.meta/`，持续迭代

### 1.3 任务类型画像聚类

**现状问题**：四级分级（Trivial/Light/Standard/Critical）粒度过粗，不同类型任务规律不同。

**优化方案**：
- 根据 `.meta/` 数据自动聚类任务类型（UI 类 / 算法类 / 重构类 / 配置类 / 文档类）
- 每类任务自动匹配最优的流程强度、模型档位、预算配额
- 比如：文档类任务普遍 1 轮 review 就过 → 自动减少默认 review 轮数
- 比如：重构类任务普遍返工多 → 自动增加预算和初始 review 强度

### 1.4 自适应价值

把工作流优化从「一次性设计」变成「持续在线学习」。模型在进化，工作流的参数也应该跟着进化，而不是每次模型升级都靠人手动改一遍配置。

## 二、失败模式分类与定向自愈

v2.1 的熔断是「一刀切」——轮数到了就停，提示人工验收。但不同的卡住原因，解法完全不同。60-70% 的卡住场景其实可以自动定向修复。

### 2.1 五类失败模式识别

| 失败模式 | 识别特征 | 自愈策略 |
|---|---|---|
| **需求不清型** | review 反复提「需求不明确」类 issue，planner 改了又改 | 自动回退到 propose 阶段，触发需求重澄清，重置计数 |
| **设计缺陷型** | 同一类设计问题反复出现，executor 在现有框架下改不动 | 自动回退到 plan 阶段重设计，重置 review 计数 |
| **执行偏差型** | spec 没问题，代码写偏了，局部性问题 | 触发 targeted review，只查偏离点，不全量重审 |
| **粒度过粗型** | 单 wave task 数爆炸，executor 上下文溢出 | 自动建议拆 wave 或拆 change，给出拆分建议 |
| **上下文腐化型** | 早期决策被后期推翻，前后矛盾，逻辑漂移 | 触发 fresh-context 重跑 planner，不带着腐烂上下文继续 |

### 2.2 自愈执行流程

```
检测到卡住（轮数超阈值 / 无进展熔断）
        │
        ▼
失败模式分类器（基于 review issue 关键词 + .meta/ 数据）
        │
        ├─ 可自愈 → 执行对应自愈策略 → 重置对应计数器 → 继续
        │
        └─ 不可自愈 → 冒泡人工介入，附带分类诊断结果
```

### 2.3 自愈置信度

- 前 10 次自愈都标记为「建议」，需要人工确认才执行
- 累计自愈成功率 > 80% → 切换为自动执行
- 某类模式连续自愈失败 2 次 → 降级为人工确认模式

## 三、Spec 生命周期治理

v2.1 解决了「代码实现偏离 spec」的问题，但没解决 **spec 自己变烂** 的问题。项目跑久了 `specs/` 目录会膨胀、矛盾、过时，变成新的腐化源。

### 3.1 Spec 置信度体系

每个需求条目带置信度标记：

```markdown
### Requirement: Session Expiration
- **Confidence**: high
- **Status**: active
- **Last verified**: 2026-07-15
SHALL: User sessions expire after 30 minutes of inactivity.
```

置信度规则：
- **high**：从代码反推、有测试覆盖、经过验证
- **medium**：写了 spec、已实现、但测试覆盖不全
- **low**：只写了 spec 没实现，或从遗留系统逆向生成、未经验证

**消费规则**：
- planner 做设计时，低置信度 spec 必须先做确认，不能直接当事实
- reviewer 审查时，低置信度 spec 不作为硬性 FAIL 标准

### 3.2 Spec 质量巡检

定期自动扫描 `specs/`：

1. **冗余检测**：语义重复的需求条目 → 提示合并
2. **矛盾检测**：互相冲突的 SHALL/MUST → 标记冲突，人工裁决
3. **折旧检测**：超过 N 个 change 没被引用过 → 标记 STALE，提示确认是否废弃
4. **覆盖率检测**：哪些 spec 条目有对应测试、哪些只是纯文档

命令：`bp spec audit`

### 3.3 Spec 演化版本管理

- 每个 spec 条目带 `since: v0.4.2` 版本标记
- archive 合并 delta 时自动记录变更历史
- `bp spec diff <version>` 对比两个版本间的 spec 变化
- 重大变更（REMOVED 核心需求）自动触发全项目影响分析

### 3.4 为什么重要

spec 不是写完就完事了。它也是代码，也需要 lint、refactor、deprecate 的完整生命周期。否则半年后 `specs/` 会变成没人敢删、也没人信的僵尸文档，反而成为新的上下文噪声源。

## 四、跨 Change 依赖与调度

v2.1 每个 change 是独立单元，但真实项目中 change 之间有依赖关系。并行开发时会踩脚、底层改了上层不知道。

### 4.1 Change 依赖声明

`proposal.md` 增加依赖声明：

```markdown
## Dependencies
- Depends On: add-auth-middleware, refactor-user-model
- Blocks: migrate-to-new-api
```

**约束规则**：
- 被依赖的 change 未 archive，当前 change 不能进入 apply
- 依赖的 change 被打回重写，当前 change 自动标记「待重新对齐」

### 4.2 Change 调度图

- `bp deps graph` 输出所有活跃 change 的依赖 DAG
- 无依赖的自动标记「可并行」
- 有依赖链的自动排序
- 类似 GSD 的 wave 概念，但上升到 change 粒度

### 4.3 级联变更检测

一个 change archive 后：
1. 自动扫描所有依赖它的活跃 change
2. 对比 delta spec，判断是否有接口/行为变更
3. 受影响的 change 自动触发「设计重审」告警
4. 防止底层改了接口，上层 change 带着旧设计一路跑到 review 才炸

### 4.4 并行冲突检测

增强 v2.1 已有的 parallel change conflict detection：
- 不仅检测是否碰同一个文件
- 检测是否修改同一个 spec 条目
- 检测是否触碰同一个模块的上下游
- 冲突级别分：无冲突 / 弱冲突（相邻模块）/ 强冲突（同文件同函数）

## 五、不确定性量化与分级验证

v2.1 的验证是二元的（PASS / FAIL / NEEDS_REVISION），但实际上 AI 产出的不同部分置信度不一样。全量统一审查既浪费钱，也容易让 reviewer 疲劳。

### 5.1 产出置信度标注

每个子代理的产出自带置信度评估：

**Planner 输出示例**：
```markdown
### DS-3: Payment Processor
- **Confidence**: medium
- **Reason**: 现有代码库支付模块结构复杂，设计基于摘要推断，可能遗漏边缘场景
```

**置信度分级规则**：
- **high**：改动小、路径清晰、有大量先例参考
- **medium**：涉及新模块、需要做架构决策、存在多种可行方案
- **low**：探索性任务、核心逻辑改动、涉及复杂状态机

### 5.2 分层验证策略

根据置信度匹配不同强度的验证：

| 置信度 | 风险级别 | 验证方式 |
|---|---|---|
| high | 低 | 自动验证（跑测试 + 静态检查），跳过人工 review |
| medium | 中 | 子代理 review，重点查设计合理性 |
| low | 高 | 子代理三重 review + 人工确认门禁 |

不是所有内容都用同一强度审查。高置信度的小改动没必要走完整三重 review，浪费 Token。

### 5.3 不确定性溯源与学习

- review 发现问题时，回溯是哪个环节的低置信度判断导致的
- 统计：哪些类型的决策 AI 容易高估自己的置信度
- 统计：哪些模块/哪些类型的改动天然就是低置信度
- 反推优化：对「经常翻车的领域」自动提高初始审查强度

## 六、工作流熵减机制（元层面）

这是最元的一层。Blueprint 加了这么多功能，框架本身会不会随着版本迭代越来越重，走向它自己反对的「官僚主义」？

**必须内置防止自身膨胀的机制。**

### 6.1 流程使用率统计

- 每个功能/步骤统计实际使用率（有多少 change 走到了这一步）
- 连续 N 个项目都没触发的功能 → 标记为 DEPRECATED
- 类似代码的 dead code 检测，但检测的是工作流步骤

### 6.2 定期瘦身审计

新增 `bp audit workflow` 命令：

1. **Prompt 冗余检测**：检查子代理 prompt 里哪些片段模型已经内化，实际不影响产出
2. **步骤必要性检测**：跳过某步骤后质量是否显著下降
3. **模型版本适配建议**：换了新模型后，哪些流程可以降级或删除

**触发时机**：
- 大版本模型升级时自动触发
- 每 50 个 change 自动跑一次
- 人工手动触发

### 6.3 复杂度预算

整个框架设硬上限：
- 子代理 prompt 总 Token 数上限
- 单个 change 的最大步骤数上限
- 配置项总数上限

加新功能必须满足其一：
- 删掉一个旧功能
- 证明旧功能确实还有不可替代的价值
- 证明新功能的收益远大于复杂度成本

类似代码的测试覆盖率门禁，但管的是**认知复杂度**。

### 6.4 为什么必须有这个

防止「为上一代模型写的兼容层永久保留」的悲剧。模型每升级一代，工作流就应该瘦一圈，而不是只做加法不做减法。熵减机制是 Blueprint 对抗自身官僚化的免疫系统。

## 七、实施路线图（v2.1 之后）

### 阶段一：闭环基础（P0）

**目标**：先把数据采集和基础自愈做了

| 改动 | 工作量 | 依赖 |
|---|---|---|
| 失败模式分类器 | 中 | 需要 v2.1 的 .meta/ 数据结构 |
| 定向自愈（需求不清/设计缺陷 两类） | 中 | 分类器 |
| 降级成功率反馈环 | 小 | v2.1 已有降级机制 |

### 阶段二：Spec 治理（P1）

**目标**：防止 specs/ 本身腐化

| 改动 | 工作量 | 依赖 |
|---|---|---|
| Spec 置信度体系 | 中 | delta spec 模板扩展 |
| Spec 质量巡检（audit 命令） | 中 | 置信度标记 |
| Spec 折旧与冗余检测 | 小 | 巡检基础上扩展 |

### 阶段三：跨 Change 调度（P2）

**目标**：支持多 change 并行的真实项目场景

| 改动 | 工作量 | 依赖 |
|---|---|---|
| Change 依赖声明与门禁 | 小 | proposal 模板扩展 |
| 级联变更检测 | 中 | 依赖图 + spec diff |
| 并行冲突检测增强 | 中 | codebase-map |

### 阶段四：自适应进化（P3）

**目标**：从静态配置到动态自调优

| 改动 | 工作量 | 依赖 |
|---|---|---|
| 分级阈值自动校准 | 中 | 足够的历史数据 |
| 任务类型画像聚类 | 大 | 历史数据 + 分类算法 |
| 不确定性量化 + 分级验证 | 大 | 置信度标注体系 |

### 阶段五：元治理（P4）

**目标**：框架自身的熵减机制

| 改动 | 工作量 | 依赖 |
|---|---|---|
| 流程使用率统计 | 小 | .meta/ 数据 |
| 工作流瘦身审计 | 中 | 使用率统计 + A/B 测试框架 |
| 复杂度预算门禁 | 小 | 架构层面约束 |

## 八、衡量成功的新增指标

在 v2.1 的成本/质量/体验指标基础上，增加：

### 自适应指标
- **自愈成功率**：自动修复的卡住任务中，真正解决问题的比例 → 目标 > 70%
- **人工干预率**：需要人工介入的任务占比 → 目标持续下降
- **参数自动校准次数**：系统自己调整了多少阈值/配置

### Spec 健康度指标
- **Spec 覆盖率**：有测试对应/有代码实现的 spec 占比 → 目标 > 80%
- **Spec 折旧率**：STALE 状态的 spec 占比 → 目标 < 15%
- **Spec 冲突数**：检测到的矛盾/冗余条目数 → 目标趋近于 0

### 框架健康度指标
- **工作流步骤使用率**：每个步骤被实际使用的比例 → 目标 > 60%
- **Prompt 膨胀率**：每个版本 prompt 长度变化 → 目标：大模型升级后 prompt 长度净减少
- **复杂度预算使用率**：当前复杂度占预算上限的比例 → 目标 < 80%

## 最终原则

> **工作流框架本身也必须遵守自己倡导的工程纪律。**

- 只加有数据支撑的功能，不加感觉有用的功能
- 模型每升级一代，工作流就应该瘦一圈
- 防腐化的对象不只是代码和 spec，也包括工作流自身

Blueprint 的终极形态不是「功能最全的 AI 开发框架」，而是「**刚好够用、随模型自动进化、永远不官僚**」的最小约束系统。
