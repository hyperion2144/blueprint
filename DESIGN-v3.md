# Blueprint v3 优化方案：遥测驱动的框架进化

**版本**：v2.0  
**日期**：2026-07-22  
**基于版本**：Blueprint v2.1 + v2.1 补充优化空间

---

## 核心判断

v2.1 完成了"按风险定价流程强度"。v3 的方向是**遥测驱动的框架进化**：

> **工作流自进化不是用户执行的功能，与使用 bp 的用户无关。用户端只做遥测采集；进化由 bp 项目维护者基于遥测数据离线分析后，发布新版本触发。**

架构：

```
用户项目（bp CLI 运行时）
  ├─ 遥测采集（.meta/ + 失败模式标记 + 使用率统计 + 置信度标注）
  ├─ 匿名化处理（脱敏代码片段、路径、项目名）
  └─ opt-in 上传 / 手动导出（bp telemetry export）
       │
       ▼
维护者端（离线分析）
  ├─ 聚合多项目遥测数据
  ├─ 发现模式（哪些流程使用率低、哪些降级成功、哪些失败模式常见）
  ├─ 更新默认配置 / prompt / 流程强度
  └─ 发布新版本
       │
       ▼
用户端（bp update 获取进化后的版本）
```

与 v2.1 原稿的区别：
- v2.1 原稿让系统**运行时自动**校准/自愈/瘦身 → v3 改为**遥测采集 + 维护者离线分析 + 版本发布**
- 用户项目不做自动决策（更安全、更可控）
- 进化路径 = 发新版本（符合 CLI 工具的进化方式）

## 一、遥测数据采集与上传

### 1.1 自动采集（命令执行时）

用户执行 bp 命令时**自动采集**阶段数据，无需手动触发。每个命令的 handler 在执行后自动调用 `appendRunMeta()` 写入 `.meta/`。

采集时机：

| 命令 | 自动写入 | 数据 |
|---|---|---|
| bp plan | .meta/planner-run-N.json | 耗时、模型、产出校验结果 |
| bp apply | .meta/executor-wave-N.json | wave 数、task 数、验证结果 |
| bp review | .meta/reviewer-run-N.json | issue 数、分类、verdict |
| bp archive | .meta/summary.json | 汇总（总轮数、总成本、final verdict）|
| bp continue | .meta/step-usage.json | 步骤使用率（哪个 step 被触发）|

v2.1 的 .meta/ 写入靠工作流 prompt 提示（编排者手动写）。v3 改为**命令代码自动写**——handler 执行后自动记录，用户无感。

### 1.2 匿名化处理

遥测数据上传前必须脱敏：

- **代码片段**：只记 hash + 行号范围，不记原文
- **文件路径**：保留相对路径结构（`src/core/config.ts`），去项目名
- **敏感字段**（model response、error message）：标记 `redacted`
- **配置值**：只记配置项名 + 值类型（number/string/boolean），不记具体值

### 1.3 自动上报 + 手动导出

采集是自动的（命令执行时）。上报也是自动的（如果 telemetry.enabled = true）：

- **默认 opt-out**：不自动上报，.meta/ 只本地存储
- **opt-in 上报**：`config.telemetry.enabled = true` → 命令执行后异步上报（不阻塞命令，失败静默）
- **手动导出**：`bp telemetry export` 用于 opt-out 用户离线分享给维护者
- **查看状态**：`bp telemetry status` 查看采集量、是否启用上报

配置项：`config.yaml` 加 `telemetry: { enabled: false, upload_url: '' }`

上报格式：JSON 包，含 .meta/ 汇总 + 匿名化处理。异步上报失败不影响命令执行。

### 1.4 维护者端分析工具

维护者收到遥测包后，用离线工具聚合分析：

- 统计每类失败模式的频率
- 统计每个流程步骤的使用率
- 统计降级成功率按角色分布
- 统计 Spec 置信度分布
- 发现"使用率 < 阈值的步骤"（候选精简）
- 发现"成功率 < 阈值的降级"（候选收紧）

分析结果 → 更新默认配置/prompt → 发布新版本。

## 二、失败模式标记

### 2.1 五类失败模式（规则分类，非 ML）

| 失败模式 | 识别规则（基于 issue 前缀 + 频率） | 用户端行为 |
|---|---|---|
| **需求不清型** | R 类 issue 反复出现 + 同一 requirement 被提 2+ 次 | 标记 → 提示回退 propose 重澄清 |
| **设计缺陷型** | D 类 issue 反复出现 + planner design 被大改（>50% 行变化）| 标记 → 提示回退 plan 重设计 |
| **执行偏差型** | Q 类 issue 集中 + spec 本身没问题 | 标记 → 提示 targeted review（只查偏离点）|
| **粒度过粗型** | 单 wave task 数 > 8 + executor 上下文溢出 | 标记 → 提示拆 wave/change |
| **上下文腐化型** | 早期 design 决策被后期 fix 推翻 + 逻辑漂移 | 标记 → 提示 fresh-context 重跑 planner |

### 2.2 标记而非自愈

v2.1 原稿让系统自动自愈。v3 改为**标记 + 提示**：

- 用户端：检测到卡住 → 自动分类失败模式 → 在 continue 输出标记 `[FAILURE-MODE: design-flaw]` + 建议策略
- **不自动执行自愈**——用户决定是否采纳建议
- 标记写入 `.meta/failure-mode.json` 供遥测

理由：自动自愈风险高（可能错分类 → 错操作 → 更乱）。标记 + 提示更安全，且维护者可以分析标记数据改进分类规则。

### 2.3 维护者进化

维护者分析遥测的失败模式数据：
- 哪类失败模式最常见 → 对应流程步骤需要改进
- 某类模式的自愈建议采纳率 → 判断建议是否有效
- 采纳建议后是否减少卡住 → 判断建议是否正确

分析结果 → 新版本改进对应流程/prompt。

## 三、Spec 生命周期治理

### 3.1 Spec 置信度自动推断（非手动标记）

v2.1 原稿让手动标记 high/medium/low。v3 改为**自动推断**：

| 置信度 | 自动推断规则 |
|---|---|
| **high** | 有对应测试文件 + 有代码实现（从 codebase-map exports 交叉验证）|
| **medium** | 有代码实现但无测试覆盖 |
| **low** | 只有 spec 无代码实现，或从遗留系统逆向生成未验证 |

推断来源：
- codebase-map 的 public_api 列表（已实现）
- 测试文件命名模式（`*.test.ts` / `test_*.py` / `*_test.go`）
- spec 条目的 requirement 名与 exports 的匹配

### 3.2 Spec 质量巡检

`bp spec audit` 扫描 `bp/specs/`：

1. **冗余检测**：相同 requirement 名 → 提示合并（精确匹配，不做语义分析）
2. **折旧检测**：超过 N 个 change 未被 delta spec 引用 → 标记 STALE
3. **覆盖率检测**：哪些 spec 条目有对应测试 → 输出覆盖率
4. **矛盾检测**：相同 domain 下两个 SHALL 互相冲突 → 标记冲突（简化：检测 requirement 名相同但内容不同）

**不做的**：语义级矛盾检测（需要 NLP，太重）。

### 3.3 Spec 演化版本管理

- 每个 requirement 带 `since: v0.4.2` 版本标记
- archive 合并 delta 时自动记录变更历史
- `bp spec diff <version>` 对比两个版本间的 spec 变化
- 重大变更（REMOVED 核心需求）自动触发全项目影响分析（用 codebase-map queryImpact）

### 3.4 遥测：Spec 健康度

`bp spec audit` 输出写入 `.meta/spec-health.json`，供遥测上传。维护者分析：
- 哪些 domain 的 spec 最容易 STALE
- 覆盖率分布 → 哪些 domain 测试覆盖最差

## 四、跨 Change 依赖与调度

### 4.1 Change 依赖声明（已有基础，扩展）

v2.1 已有 `## Dependencies` section + continue 依赖检测。扩展：

```markdown
## Dependencies
- Depends On: add-auth-middleware, refactor-user-model
- Blocks: migrate-to-new-api
```

约束规则：
- 被依赖的 change 未 archive → 当前 change 不能进入 apply（v2.1 已实现）
- 依赖的 change 被打回重写 → 当前 change 自动标记"待重新对齐"（新增）
- **死锁检测**：A 依赖 B，B 依赖 A → 检测环 → 报错（新增）

### 4.2 Change 调度图

`bp deps graph` 输出所有活跃 change 的依赖 DAG：

```
add-auth-middleware ──┐
                      ├──→ user-profile ──→ migrate-to-new-api
refactor-user-model ──┘
```

- 无依赖的自动标记"可并行"
- 有依赖链的自动排序
- 类似 wave 概念，但上升到 change 粒度

### 4.3 级联变更检测

一个 change archive 后：
1. 自动扫描所有依赖它的活跃 change（用 codebase-map queryImpact）
2. 对比 delta spec，判断是否有接口/行为变更
3. 受影响的 change 自动触发"设计重审"告警
4. 防止底层改了接口，上层 change 带着旧设计一路跑到 review 才炸

### 4.4 并行冲突检测

增强 v2.1 已有的文件 manifest 重叠检测：
- 检测是否修改同一文件（v2.1 F3b 已有）
- 检测是否修改同一 spec 条目（新增）
- 检测是否触碰同一模块的上下游（用 codebase-map depends_on，新增）
- 冲突级别：无冲突 / 弱冲突（相邻模块）/ 强冲突（同文件同函数）

**这章是纯工程约束型，不需要遥测。**

## 五、不确定性量化与分层验证

### 5.1 产出置信度标注

子代理产出自带置信度评估：

**Planner 输出**：
```markdown
### DS-3: Payment Processor
- **Confidence**: medium
- **Reason**: 支付模块结构复杂，设计基于摘要推断，可能遗漏边缘场景
```

置信度分级规则（标注在 design.md DS-N）：
- **high**：改动小、路径清晰、有大量先例参考
- **medium**：涉及新模块、需要架构决策、存在多种可行方案
- **low**：探索性任务、核心逻辑改动、涉及复杂状态机

### 5.2 分层验证策略

根据置信度匹配验证强度：

| 置信度 | 风险级别 | 验证方式 |
|---|---|---|
| high | 低 | 自动验证（跑测试 + 静态检查），跳过人工 review |
| medium | 中 | 子代理 review，重点查设计合理性 |
| low | 高 | 子代理三重 review + 人工确认门禁 |

### 5.3 遥测：不确定性溯源

置信度标注 + review 结果写入 .meta/，供遥测。维护者分析：
- 哪些类型的决策 AI 容易高估置信度（标注 high 但 review FAIL）
- 哪些模块/类型的改动天然低置信度
- 反推：对"经常翻车的领域"在新版本自动提高初始审查强度

## 六、框架熵减（维护者端）

### 6.1 流程使用率统计（遥测采集）

每个功能/步骤统计实际使用率：
- 有多少 change 走到了这一步
- 某步骤连续 N 个项目都没触发 → 遥测标记 DEPRECATED

### 6.2 定期瘦身审计（维护者做）

`bp audit workflow`（维护者工具，基于遥测数据）：

1. **Prompt 冗余检测**：基于 v2.1 的 CAPABILITY-COMPENSATION 标记，检查哪些 section 不影响产出（遥测对比：有标记 vs 无标记的产出质量差异）
2. **步骤必要性检测**：跳过某步骤后质量是否显著下降（遥测对比）
3. **模型版本适配建议**：换了新模型后，哪些流程可以降级或删除

**触发时机**：维护者收到足够遥测数据后手动触发，不是用户端自动。

### 6.3 复杂度预算

框架设硬上限（维护者设定，用户端检查）：

| 预算项 | 计算方式 | 上限 |
|---|---|---|
| Prompt 总 token 数 | 字符数 / 4（粗估） | 50,000 |
| 单 change 最大步骤数 | schema.artifacts.length + schema.steps.length | 15 |
| 配置项总数 | ProjectConfig 字段数 | 30 |

加新功能必须满足其一：
- 删掉一个旧功能（复杂度预算不超）
- 证明旧功能确实还有不可替代的价值
- 证明新功能的收益远大于复杂度成本

### 6.4 为什么必须有这个

防止"为上一代模型写的兼容层永久保留"。模型每升级一代，维护者基于遥测数据瘦一圈框架，而不是只做加法不做减法。**熵减是 Blueprint 对抗自身官僚化的免疫系统。**

## 八、衡量成功的新增指标

### 遥测覆盖率
- **遥测导出项目数**：有多少项目贡献了遥测数据
- **遥测数据量**：累计 change 数 + .meta/ 记录数
- **失败模式标记覆盖率**：卡住场景中有失败模式标记的比例 → 目标 > 80%

### Spec 健康度
- **Spec 覆盖率**：有测试/代码的 spec 占比 → 目标 > 80%
- **Spec 折旧率**：STALE spec 占比 → 目标 < 15%
- **Spec 冲突数**：矛盾/冗余条目数 → 目标趋近 0

### 框架健康度（维护者端）
- **流程使用率**：每个步骤被实际使用的比例 → 目标 > 60%
- **Prompt 膨胀率**：每版本 prompt 长度变化 → 目标：大模型升级后净减少
- **复杂度预算使用率**：当前复杂度占预算的比例 → 目标 < 80%
- **失败模式建议采纳率**：用户采纳失败模式建议的比例 → 目标 > 50%

## 最终原则

> **工作流框架本身也必须遵守自己倡导的工程纪律。**

- 只加有遥测数据支撑的功能，不加感觉有用的功能
- 自进化是维护者的事，不是用户的——用户只采集，维护者才进化
- 模型每升级一代，维护者基于遥测数据瘦一圈框架
- 防腐化的对象不只是代码和 spec，也包括工作流自身

Blueprint 的终极形态不是"功能最全的 AI 开发框架"，而是"**刚好够用、遥测驱动进化、永远不官僚**"的最小约束系统。
