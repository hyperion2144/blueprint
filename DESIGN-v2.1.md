# Blueprint 框架优化方案：模型增强时代的工作流再设计

**版本**：v1.0  
**日期**：2026-07-21  
**适用版本**：Blueprint v0.6 / v2  

---

## 核心结论

Blueprint v2 已经走在了正确方向上（砍掉状态机官僚主义、保留工程严谨性），但在**模型能力持续增强**的背景下，需要从「固定强度的工作流」进化为「**按风险动态匹配强度的工程约束系统**」。

核心原则：
> **越依赖"教模型思考"的流程，折旧越快；越接近团队工程政策的约束，生命周期越长。**

---

## 一、判断框架：两类流程的价值分化

模型升级后，工作流中的元素会分化为两类，价值走向完全相反：

| 流程类型 | 解决的问题 | 模型增强后的价值趋势 |
|---|---|---|
| **能力补偿型** | 教 Agent 如何澄清、拆解、规划和反思 | 边际价值持续下降，模型原生能力增强后成为冗余 |
| **工程约束型** | 定义验收、审批、验证、Review 和停止条件 | 价值稳定，不随模型升级自动消失，甚至更重要 |

**Blueprint 的优化方向**：保留并强化工程约束型流程，裁剪并按需启用能力补偿型流程。

---

## 二、Blueprint 现状评估

### 2.1 已经做对的部分

| 设计决策 | 价值判断 | 说明 |
|---|---|---|
| 砍掉状态机、25个命令、4层嵌套 | ✅ 正确 | 这些是典型的"官僚主义"，不产生工程价值 |
| 保留结构化 design/tasks（DS-N、D-N、wave） | ✅ 正确 | 结构化契约是工程约束，子代理之间的协作接口 |
| 三子代理架构（planner/executor/reviewer） | ✅ 正确 | fresh-context 防御 context rot，是架构级价值 |
| delta specs（ADDED/MODIFIED/REMOVED） | ✅ 正确 | 需求对齐机制，brownfield 友好 |
| spec 门控 + 质量门控 | ✅ 正确 | 验收标准，属于工程约束 |
| lite/standard 双 profile | ⚠️ 方向对，粒度太粗 | 只有两档，缺少中间的风险触发型档位 |
| lightweight/full 自动分类 | ⚠️ 维度单一 | 仅按 task type 判断，没有考虑失败代价 |

### 2.2 当前架构的核心缺口

**缺少"比例控制层"**——这是 Superpowers 1.2 亿 Token 事件暴露的核心问题，Blueprint 目前同样存在：

1. 没有任务粒度准入检查（多大的 change 该拆）
2. 没有子代理派发次数上限
3. 没有 review-fix 循环轮数限制
4. 没有时间/Token 预算概念
5. 没有收益递减判断（什么时候该停，什么时候该人工介入）

结果就是：**工作流保持活跃，任务却没有稳定收敛**。

## 三、优化空间（按优先级排序）

### P0：成本与收敛控制（防止失控）

**优先级最高**。模型越强、执行越自主，失控的成本越高。

#### 3.1.1 Change 级预算配置

在 `config.yaml` 中增加预算控制项：

```yaml
# bp/config.yaml
budget:
  # 子代理派发次数上限
  max_subagent_runs: 5
  # review-fix 循环最大轮数
  max_review_rounds: 3
  # 墙钟时间上限（分钟）
  max_wall_time_min: 60
  # 预估 Token 软上限（告警用）
  estimated_token_cap: 500000
  # 连续无实质进展轮数后熔断
  no_progress_fuse_rounds: 2
```

#### 3.1.2 收益递减熔断机制

每轮 review 自动统计：
- 新增 issue 总数
- 阻塞性 issue（BLOCKER）数量
- 同类重复 issue 占比

**熔断触发条件**：连续两轮新增 issue < 阈值 且 无阻塞性问题 → 自动结束，输出「边际收益不足，建议人工验收」提示。

#### 3.1.3 任务粒度准入检查

`bp plan` 阶段自动检查并告警：
- tasks 总数 > 20 → 建议拆分 change
- 涉及文件数 > 15 → 建议拆分 change
- wave 数 > 5 → 建议拆分 change
- 单个 wave task 数 > 8 → 建议拆分为多个 wave

**设计原则**：不是强制阻止，而是给出明确的风险提示和建议，由用户决策。

### P1：任务风险分级路由（按风险定价流程）

从当前的 lite/standard 二元切换，扩展为**四级动态路由**。

#### 3.2.1 四级任务分级体系

| 级别 | 触发条件 | 执行方式 | 控制强度 | 典型场景 |
|---|---|---|---|---|
| **Trivial** | 单文件、纯 docs/config/scaffolding、无行为变更 | 主对话 inline 执行，不派发子代理 | 仅验证产物格式 | 改个配置、补个文档、加个类型定义 |
| **Light** | 2-5 文件、低风险行为变更、测试覆盖良好 | 单 agent 串行执行 + 轻量验证 | 跳过 TDD 强制，review 可选 | 小功能迭代、UI 微调、非核心路径改动 |
| **Standard** | 跨模块、新增行为、中等风险 | planner + wave executor + 三重 review | TDD 强制 + spec 门控 | 新功能开发、核心模块重构 |
| **Critical** | 涉及权限/资金/数据一致性、核心路径 | 完整流程 + 安全审计 + 人工审批点 | 双 reviewer + 人工确认门禁 | 鉴权改动、支付逻辑、数据迁移 |

#### 3.2.2 自动分级 + 人工覆盖

- `bp propose` 阶段根据 scope 自动评估级别，给出建议
- 用户可通过 `--level standard` 显式指定，覆盖自动判断
- 分级结果写入 `proposal.md` 头部，后续流程据此调整强度

#### 3.2.3 分级对应的流程差异

| 流程元素 | Trivial | Light | Standard | Critical |
|---|---|---|---|---|
| Planner 子代理 | ❌ | ⚠️ 简化版 | ✅ 完整 | ✅ 完整 + 安全维度 |
| TDD 强制 | ❌ | ❌ | ✅ behavior 类 | ✅ 全部 |
| Wave 并发 | ❌ 串行 | ❌ 串行 | ✅ | ✅ |
| Review 子代理 | ❌ | ⚠️ 可选 | ✅ 三重 | ✅ 三重 + 安全审计 |
| Spec 门控 | ❌ | ⚠️ 仅检查 | ✅ 强制 | ✅ 强制 + 人工复核 |
| Archive 前人工确认 | ❌ | ❌ | ❌ | ✅ |

### P2：模型分档与智能选型

#### 3.3.1 按角色分档（深化现有配置）

当前 `models` 配置只是静态映射，需要明确每个角色的模型选型策略：

| 子代理角色 | 推荐模型档位 | 理由 |
|---|---|---|
| Planner | 强推理档（slow/thinking） | 设计错了后面全错，值得投入 |
| Executor（复杂 wave） | 平衡档（standard） | 按图施工，不需要最强推理 |
| Executor（简单 wave） | 快速档（fast） | 机械性实现，快且省 |
| Reviewer | 强推理档（slow） | 找问题需要深度理解 |
| Reviewer（第二轮后） | 平衡档（standard） | 第二轮主要是验证修复，难度下降 |

#### 3.3.2 动态降级机制

- 任务分级为 Trivial/Light → planner 自动降级为 fast 模型
- 第一轮 review 未发现严重问题 → 第二轮 review 自动降级
- 同类 task 连续成功执行 → 后续 executor 自动降档

**参考数据**：OpenAI 内部评估显示，精简 prompt + 合理选型可降低 33-67% 成本，同时提升 10-15% 评分。

---

### P3：能力补偿型流程的裁剪

#### 3.4.1 保留清单（工程约束，不能删）

这些是子代理之间的结构化契约，或者是验收标准，属于工程约束型：

- DS-N 组件分解格式要求 → executor/reviewer 要消费
- D-N 决策记录格式 → 可追溯性
- Wave 依赖标注规则 → 执行调度需要
- spec_ref 关联要求 → reviewer 做 spec 门控
- GIVEN/WHEN/THEN 场景格式 → 测试与验收的接口
- Review 三维度（spec/quality/goal）结构 → 验收标准

#### 3.4.2 可精简清单（能力补偿，模型已内化）

这些是"教模型怎么干活"的指导，新一代模型已经掌握，prompt 中重复描述反而增加噪声：

- 需求澄清的步骤指导（模型已经会问澄清问题）
- 任务拆解的方法论描述（模型已经知道怎么拆）
- "先思考再输出"这类通用套话
- 基础 TDD 概念解释（什么是红绿绿，模型已经知道）
- 代码规范的重复描述（通过 conventions 文件注入一次即可，不要每个子代理 prompt 都复述）

#### 3.4.3 验证方法

做 A/B 对比测试：
1. 选取 10 个代表性任务
2. 分别用完整版 prompt 和精简版 prompt 跑
3. 对比产出质量（design 完整性、task 覆盖率、review 通过率）
4. 对比成本（Token 数、执行时间）

**判断标准**：质量差异 < 5% 但成本下降 > 20% → 大胆精简。

### P4：上下文工程优化

#### 3.5.1 按需加载 Specs

**现状问题**：planner 可能读取全部 `specs/` 目录，但一个 change 通常只涉及 1-2 个 domain。

**优化方案**：
- 根据 `proposal.md` 的 Scope 预筛选相关 domain
- 只注入相关 spec 文件，不相关的完全不加载
- 预估可减少 planner 上下文 30-60%

#### 3.5.2 Executor 上下文瘦身

**现状问题**：每个 wave 的 executor 都拿到完整的 design.md + tasks.md。

**优化方案**：
- 只注入当前 wave 的 tasks
- 只注入与当前 wave 相关的 design 片段（按 DS-N 引用筛选）
- 对应 GSD 的 anchor pruning 思路：每个子代理只拿自己需要的那部分

#### 3.5.3 代码库摘要预生成

**现状问题**：planner 需要理解代码库现状，但全量读取成本太高。

**优化方案**：
- 预生成 codebase map（模块结构、关键接口、依赖关系）
- planner 先看摘要，按需深入具体文件
- 类似 GSD 的 codebase-mapper 角色，但更轻量

### P5：可观测性与反馈闭环

#### 3.6.1 执行元数据记录

每个 change 增加 `.meta/` 目录，记录执行过程数据：

```
changes/<name>/.meta/
├── planner-run-1.json      # 耗时、Token 估算、模型档位、产出校验结果
├── executor-wave-1.json    # 每个 wave 的执行数据
├── executor-wave-2.json
├── review-round-1.json     # 每轮 review 的 issue 数、分类、严重程度
└── summary.json            # 汇总：总耗时、总轮数、总成本估算
```

#### 3.6.2 质量回溯指标

每个 change archive 时自动统计：
- Review 发现的 issue 总数及分类（spec 偏离 / 质量问题 / 目标偏差）
- 收敛轮数（几轮 review 才通过）
- 哪些类型的 task 最容易出问题
- Planner 遗漏率（review 发现的 planner 阶段该考虑到的问题）

**长期价值**：积累数据后，可以反推：
- 哪些类型的任务 planner 容易漏设计
- 哪些类型的任务 executor 容易出错
- 不同级别的任务实际需要几轮 review
- 形成数据驱动的工作流持续优化

#### 3.6.3 成本与质量仪表盘

新增 `bp stats` 命令：
- 查看单个 change 的执行数据
- 查看项目级汇总（平均成本、平均收敛轮数、各级别任务分布）
- 让团队能直观回答"这套工作流花了多少钱、带来了多少质量收益"

## 四、实施路线图

### 阶段一：止血（防止失控）

**目标**：先解决最容易出问题的成本失控

| 改动 | 工作量 | 预期收益 |
|---|---|---|
| 增加 max_review_rounds 限制 | 小 | 防止无限 review 循环 |
| 增加任务粒度告警 | 小 | 防止 change 过大导致收敛困难 |
| 增加 no_progress_fuse 熔断 | 中 | 自动识别边际收益递减场景 |

### 阶段二：分级（按风险定价）

**目标**：让流程强度和任务风险匹配

| 改动 | 工作量 | 预期收益 |
|---|---|---|
| 四级任务分级体系 | 中 | Trivial/Light 任务成本下降 50%+ |
| 自动分级 + 人工覆盖 | 中 | 降低使用门槛 |
| 分级对应的流程差异实现 | 大 | 完整的动态强度体系 |

### 阶段三：精简（裁剪冗余）

**目标**：去掉模型已经内化的能力补偿

| 改动 | 工作量 | 预期收益 |
|---|---|---|
| Planner prompt 审计与精简 | 中 | Token 减少 20-30%，质量不下降 |
| Executor/Reviewer prompt 精简 | 中 | 进一步降低单轮成本 |
| A/B 测试验证框架 | 小 | 建立持续优化的验证方法 |

### 阶段四：智能（数据驱动）

**目标**：从静态配置进化为动态优化

| 改动 | 工作量 | 预期收益 |
|---|---|---|
| 执行元数据记录 | 中 | 建立可观测性基础 |
| 模型分档自动选型 | 中 | 进一步降低成本 15-25% |
| 上下文按需加载 | 大 | 大幅减少子代理上下文成本 |
| 质量回溯与统计仪表盘 | 中 | 数据驱动的持续优化能力 |

## 五、衡量成功的指标

优化是否有效，用这几个指标衡量：

### 成本指标
- **单 change 平均 Token 消耗**：目标下降 30-50%
- **子代理派发次数分布**：Trivial/Light 任务不再派发重型子代理
- **收敛轮数中位数**：review-fix 循环平均轮数下降

### 质量指标
- **Review 通过率（首轮）**：不下降甚至提升（因为 prompt 更聚焦）
- **Spec 偏离率**：保持稳定或下降（工程约束没有削弱）
- **线上回归率**：终极指标，保持稳定

### 体验指标
- **小任务完成时间**：Trivial/Light 级任务明显变快
- **用户手动干预率**：不需要频繁手动 stop 失控任务
- **流程感知度**：用户感觉"流程在该出现的时候出现，不该出现的时候不打扰"

## 六、最终原则

> **模型升级，工作流也要重新定价。**

团队应该保留能够控制真实工程风险的部分，删掉只在补偿上一代模型的步骤。否则，旧模型的缺陷会被固化成团队永久 SOP——模型越来越强，开发却继续为过时的兼容层付费。

Blueprint 的进化方向应该是：**流程更短，但验收更硬；默认更轻，但风险点更严。**

## 七、补充优化空间

> 以下内容基于对 v2.1 方案的深化分析，分三类：现有方案增强、新增维度、代码库摘要深化。

### 7.1 现有方案增强（8 点）

#### 7.1.1 预算消耗可视化（增强 P0）

v2.1 提了 budget 配置项，但缺消耗进度感知。建议：
- **阈值分级**：80% 软告警（继续但提示）、100% 硬停止
- **per-change 而非 per-project**：不同 change 风险不同，Critical change 应给更多预算
- **消耗可视化**：每轮 sub-agent 完成后输出当前消耗百分比

#### 7.1.2 熔断后人工恢复流程（增强 P0）

v2.1 说"连续两轮新增 issue < 阈值 → 自动结束，建议人工验收"，但没提验收后怎么恢复。补：
- 人工验收 PASS → 直接 archive（跳过剩余 review 轮数）
- 人工验收发现新问题 → \`bp apply --fix\` 重入循环（重置 max_review_rounds 计数）
- 人工验收结果写入 review.md 的 \`## Human Verdict\` section

#### 7.1.3 四级分级申诉机制（增强 P1）

自动分级可能判错（把 Critical 判成 Light）。补：
- **reviewer 可升级**：发现实际风险高于 proposal 标注级别时，review.md 加 \`Level Escalation: Light→Critical\` 标记，触发完整流程重走
- **executor 可降级申诉**：发现任务比预期简单，标记 \`Level Suggestion: Standard→Light\` 供 reviewer 参考
- **分级申诉记录**：写入 .meta/ 供后续优化自动分级算法

#### 7.1.4 动态降级回退机制（增强 P2）

v2.1 说"第一轮 review 无严重问题 → 第二轮降档"，但降档后发现 BLOCKER 怎么办。补：
- 降档后发现 BLOCKER → 自动升级回原档位 + 标记 \`degradation_failed\`
- 连续 2 次 \`degradation_failed\` → 禁止该角色自动降级（需人工解锁）
- 降级失败记录写入 .meta/，反推哪些角色不宜自动降档

#### 7.1.5 prompt 精简安全网（增强 P3）

v2.1 建议 A/B 测试，但缺回退路径。补：
- 完整版 prompt 保留为 \`--full-prompt\` 选项，精简版为默认
- 精简版产出质量连续 3 次 < 完整版 → 自动切回完整版 + 告警
- config 加 \`prompt_profile: lite|standard|full\` 三档可切，按任务级别自动匹配

#### 7.1.6 上下文大小预算（增强 P4）

v2.1 提了按需加载，但没给子代理上下文大小预算。补：

| 角色 | 上下文预算 | 说明 |
|---|---|---|
| planner | 32 KB | 含 L0+L1 摘要 + 相关 domain specs + proposal |
| executor | 8 KB | 只含当前 wave 的 tasks + 相关 DS-N design 片段 |
| reviewer | 16 KB | 含全部 artifacts + diff + 相关 specs |

超预算时 spec-injectator 截断或分页，而非整文件注入。

#### 7.1.7 .meta/ 隐私安全处理（增强 P5）

planner-run-N.json 可能含代码片段、路径、API key。补：
- \`.meta/\` 加入 \`.gitignore\`（或 config 加 \`meta.commit: false\`）
- 脱敏：代码片段只记 hash + 行号范围，不记原文
- 敏感字段（model response、error messages）标记 \`redacted\`

#### 7.1.8 跨 change 数据聚合与基准对比（增强 P5）

v2.1 的 bp stats 是 per-change + per-project。扩展：
- **趋势图**：review 通过率/收敛轮数随时间变化（是否在改善）
- **基准对比**：同类型任务的平均成本，供异常检测
- **异常告警**：某 change 成本/轮数 > 项目中位数 2σ → 标记异常，建议人工审查

### 7.2 新增维度（5 点）

#### 7.2.1 变更影响分析（Impact Analysis）

v2.1 未提。plan 阶段自动分析"这个 change 影响哪些模块/调用方/测试"——属工程约束型（不随模型折旧）。可复用 LSP \`references\` 做静态分析，产出 design.md 的 \`## Impact Analysis\` section。

#### 7.2.2 spec 漂移主动检测

v2.1 提了"spec 偏离率"（review 被动发现），但缺**主动检测**——定期跑 \`bp spec refresh\` 对比 spec 与代码，发现漂移告警。这是预防性而非反应性。建议 archive 后自动触发漂移检测。

#### 7.2.3 工作流版本化

v2.1 提了"流程折旧"但无版本化机制。config 加 \`workflow_version: "2.1"\`，不同版本对应不同流程强度。模型升级时切换到更轻版本，旧版本保留兼容。配合 7.1.5 的 prompt_profile 形成完整的"流程强度可调"体系。

#### 7.2.4 CI/CD 集成

v2.1 完全是交互式视角。Blueprint 的门控应支持 \`--ci\` 非交互模式：
- \`bp review --ci\`：自动跑，无人工确认，FAIL 直接 exit 1
- \`bp archive --ci\`：PASS 后自动归档
- 按级别路由：Critical 强制人工，Trivial 自动通过

#### 7.2.5 多人协作维度

v2.1 是单人视角（一个 AI + 一个用户）。实际团队场景：
- 多编排者并行操作不同 change → 需 change 锁机制（\`bp lock\`/\`bp unlock\`）
- reviewer 轮换 → 避免同一人连续 review（盲点固化），.meta/ 记录 reviewer identity
- Critical 级审批权限分级 → 不是所有人都能 approve，config 加 \`approvers: [user-list]\`

### 7.3 代码库摘要预生成深化（6 点）

#### 7.3.1 三层分级摘要

v2.1 说"模块结构、关键接口、依赖关系"混在一起。建议三层：
- **L0 项目概览**（~200 tokens）：技术栈、入口点、目录结构
- **L1 模块清单**（~1KB/模块）：每个模块的职责 + 公共 API 签名 + 依赖谁
- **L2 接口细节**（按需）：完整类型定义、函数签名

planner 先注入 L0+L1，需要实现细节时才深入 L2 或读源文件。

#### 7.3.2 版本标记 + 过时检测

摘要必须带 \`git_hash\` 字段。planner 读摘要时先比对 \`git_hash !== HEAD\` → 标记过时 → 触发增量更新，而非盲目信任旧摘要。

#### 7.3.3 增量更新机制

v2.1 没提更新策略。全量重扫成本高。建议：
- 首次：全量扫描（复用 codebase-scanner 逻辑）
- 后续：\`git diff <last_scan_hash>..HEAD\` → 只重扫变更文件 + 受影响模块的依赖链
- 触发时机：archive 后自动更新；或 planner 发现过时时按需触发

#### 7.3.4 与 specs/ 维度区分

Blueprint 已有 codebase-scanner 产出 \`bp/specs/\`（行为契约）。代码摘要和 specs/ 是**不同维度**，不能混淆：
- \`specs/\` = 系统做什么（行为约束）→ reviewer 消费
- \`codebase-map\` = 系统怎么组织（结构地图）→ planner/executor 消费

两者互补。摘要存 \`bp/.codebase-map.json\`（结构化）+ \`bp/.codebase-map.md\`（子代理可读），不污染 specs/。

#### 7.3.5 多消费者

v2.1 只说"planner 先看摘要"。但 executor 和 reviewer 也需要：
- **executor**：看当前 wave 涉及模块的摘要，知道依赖谁、被谁依赖
- **reviewer**：看 diff 涉及模块的摘要，判断改动是否波及下游

建议摘要写入 context.jsonl，按 wave 的 DS-N 引用筛选注入对应子代理。

#### 7.3.6 LSP 生成依赖图

v2.1 说"类似 GSD 的 codebase-mapper"（正则/AST 扫描）。Blueprint 可更准：用 LSP \`references\`/\`definition\` 自动生成模块间依赖关系图。对 TS/Python/Go 等 LSP 支持的语言，比正则准确。LSP 不可用时降级为 AST 解析。

配合 7.2.1 变更影响分析：planner 从摘要知道 change 涉及哪些模块，从依赖图找到受影响下游模块，在 design.md File Manifest 标注 \`[IMPACT: downstream-module-X]\`。
